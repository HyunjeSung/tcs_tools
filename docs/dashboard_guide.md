# tc_dashboard 사용법

브라우저에서 system_log TC를 실행하고 결과를 확인하는 웹 대시보드(`tools/tc_dashboard/`) 사용법이다. 설치/실행 방법은 [install.md](install.md) 참고 — 이 문서는 뜬 화면을 어떻게 쓰는지만 다룬다.

## 화면 구성

```
┌─────────────────────────────────────────────┐
│ system_log TC Dashboard   ● DUT ...  [SSH]   │ ← 헤더: 연결 상태 LED + 채널 배지
├───────────────┬───────────┬───────────┬──────┤
│ 전체 케이스   │ PASS      │ FAIL      │ 최근 실행 │ ← 요약 카드 4개
├───────────────┴───────────┴───────────┴──────┤
│ TC 실행        │ 실행 로그                     │
│ (버튼 목록)    ├───────────────────────────────┤
│                │ 결과 현황판 (최신 케이스별)     │
│                ├───────────────────────────────┤
│                │ 실행 이력 (+ 페이지네이션)     │
└───────────────┴───────────────────────────────┘
```

## 1. 헤더 — 연결 상태 LED / 채널 배지

- **초록/빨강 점**: DUT와 실제로 연결이 됐는지를 나타낸다. **TC의 PASS/FAIL과는 무관** — SSH/시리얼 연결 자체의 성공 여부만 의미한다.
  - 페이지 로드 시 + 20초마다 ICMP ping으로 갱신(전원/네트워크 생존만 확인, SSH는 절대 안 건드림 — 실패 누적 시 lockout 위험 때문)
  - TC 실행이 끝나면 그 실행의 실제 연결 결과(exit_code)로 다시 갱신됨
  - 실행 이력에서 과거 run을 클릭해서 봐도 그 run의 연결 결과로 갱신됨
- **채널 배지(SSH / SERIAL)**: 클릭하면 전환된다. **다음 실행에 사용할 채널**을 고르는 것이지, 지금 연결 상태를 보여주는 게 아니다.
  - 기본값은 항상 `SSH` — 새로고침하면 초기화된다(기억 안 함)
  - `SERIAL`로 실행하려면 TC 버튼을 누르기 전에 매번 클릭해서 전환해야 한다
  - 시리얼은 SSH가 lockout됐을 때(3회 이상 로그인 실패) 우회 수단으로 쓴다 — COM 포트가 물리적으로 연결돼 있어야 함

## 2. 요약 카드

| 카드 | 의미 |
|---|---|
| 전체 케이스 | 지금까지 실행된 케이스 중 최신 결과가 남아있는 고유 케이스 수 (결과 현황판 행 수와 동일) |
| PASS / FAIL | 그중 PASS/FAIL 개수 |
| 최근 실행 | 가장 최근에 갱신된 케이스의 시각 |

과거 run을 골라 보고 있을 때(아래 실행 이력 클릭 상태)는 카드가 그 run으로 바뀌지 않는다 — 카드는 항상 "케이스별 최신 상태" 전체 요약이다.

## 3. TC 실행 버튼

왼쪽 목록의 버튼을 누르면 해당 TC(또는 TC 세트)가 곧바로 시작된다. 실행 중에는 모든 버튼이 비활성화되며, 동시에 두 개를 실행할 수 없다(이미 실행 중이면 새 요청은 거부됨).

| 버튼 | 내용 | 비고 |
|---|---|---|
| 전체 실행 | TC01,02,03,04,05,06,07,08,09,12,13 순차 실행 | TC04 대기 포함, 수 분 소요 |
| TC04 대용량 journal timeout | TC04만 단독 | |
| TC05 압축 | TC05-4만 단독 | TC05-1~3은 setup이 필요해 전체 실행에서만 확인됨 |
| TC10-pre / TC10-post | reboot 시나리오 | pre 실행 후 **DUT가 재부팅**됨 — 부팅 완료를 기다렸다가 post 실행 |
| TC11 nmon 업로드 | happy path | BlobUploadDirector 5분+30초 대기 포함 |
| TC12 nmon retention | 단독 | |
| TC13 nmon no-op | 단독 | |
| TC11+12+13 일괄 | nmon 세트 | |
| TC14 RTC 동일 시작 병합 | 단독 | system_log 프로세스 kill 수반 |

버튼 아래 노란 글씨(있는 경우)는 실행 전 알아둬야 할 주의사항이다.

## 4. 실행 로그 (실시간)

TC 실행 중 오른쪽 위 "실행 로그" 패널에 진행 상황이 실시간으로 올라온다. 1.5초 간격으로 갱신된다.

- **SSH 채널**: 원격 명령 stdout이 그대로 흘러 들어와 항상 깔끔하다.
- **시리얼 채널**: 실제 시리얼 콘솔 내용을 그대로 실시간으로 보여준다. 물리 회선 특성상 SSH보다 지저분할 수 있고(주변 애플리케이션 로그가 섞이거나, 드물게 라인 노이즈), TC 실행 결과 요약이 실행 도중 한 번(실시간) + 완료 시 한 번(최종 정리본) 총 두 번 나올 수 있다 — **판정(PASS/FAIL 개수)은 항상 마지막 정리본 기준으로 정확하게 집계**되므로 눈에 보이는 중복은 신경 쓰지 않아도 된다.

## 5. 결과 현황판

케이스(TCxx-n) 단위로 **가장 최근에 실행된 결과**만 한 줄씩 보여준다. 어떤 run에서 나온 결과인지(Run ID)와 시각도 함께 표시된다.

- 아무 run도 선택 안 한 상태 = 전체 케이스의 최신 상태 (기본 화면)
- 실행 이력에서 특정 run을 클릭하면, 그 표가 **그 run에서 실행된 케이스만** 보여주도록 바뀐다 (상단 라벨이 `(선택한 Run ID 기준)`으로 바뀜)

## 6. 실행 이력

지금까지 실행한 run 목록을 최신순으로 10개씩 보여준다(하단 이전/다음 버튼으로 더 보기). 오래된 run은 50개를 넘으면 자동으로 정리된다.

| 열 | 의미 |
|---|---|
| Run ID | 실행 시각+대상으로 자동 생성된 ID |
| 대상 | 실행한 TC(세트) 이름 |
| 채널 | 그 run이 SSH/SERIAL 중 무엇으로 실행됐는지 |
| 상태 | `pass`/`fail`/`error`/`timeout`/`rebooted`/`interrupted` 등 |
| PASS/FAIL | 케이스 집계 |
| 시작/종료/소요시간 | |
| 결과 | **다운로드** 링크 — 클릭하면 그 run의 결과보고서 PDF를 받는다 |

행을 클릭하면 그때 실행 로그와 결과 현황판이 해당 run 기준으로 바뀐다(연결 상태 LED도 함께 갱신됨).

## 7. 결과 다운로드

완료된 run은 실행 이력 표 맨 오른쪽 "다운로드"를 누르면 PDF 결과보고서를 받을 수 있다(`tc_system_log_result_<run_id>.pdf`). TC별 PASS/FAIL 표 + 근거(output.log, journald 발췌)가 포함된다. 실행 중인 run은 다운로드 링크가 비활성화된다("-" 표시).

## 8. 자주 겪는 상황

| 증상 | 원인/대처 |
|---|---|
| 버튼이 다 비활성화됨 | 이미 다른 run이 실행 중. 실행 로그에서 진행 상황 확인, 끝날 때까지 대기 |
| 실행 실패: "이미 실행 중인 run" | 동시에 2개 못 돌림 — 정상 동작 |
| 연결 LED가 빨강 | ping 실패(전원/네트워크) 또는 최근 실행 연결 실패. 실제 TC를 돌려봐야 SSH/시리얼 가능 여부가 확정됨 |
| 시리얼 실행 시 "Access denied"(COM 포트) | 다른 프로세스가 포트를 물고 있거나 직전 세션이 아직 안 풀린 상태. 잠깐 뒤 재시도 |
| 결과가 `error`인데 PASS/FAIL 숫자가 0/0 | 실행 자체는 됐지만 결과를 못 받아온 경우(시리얼 노이즈 등) — 재실행 필요 |

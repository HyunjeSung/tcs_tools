# TC 결과를 어떻게 믿을 수 있는가

TC(Test Case)가 "PASS"라고 나왔을 때, 그걸 증명하는 흐름은 이렇다.

```
① TC 스펙 문서        ② Test Code 실행        ③ 근거 로그로 증명
(무엇을 확인할지 약속)   (스펙대로 절차 수행)      (SM / SL / 리눅스 명령어 결과)
```

핵심은 마지막 단계다. "확인했습니다"라는 **말**로 끝내지 않고, 확인에 실제로 쓴
**명령어와 그 출력**을 그대로 남긴다.

## ① 스펙 문서

`tcs/<app>/tc_<app>.md`가 "무엇이면 통과인지"를 미리 정의한다. 예 (TC01):

> 생성된 `.log.xz` 파일명이 `systemlog_{14자리}_{14자리}.log.xz` 형식이고,
> 시작 시각 ≤ 저장 시각이면 PASS.

## ② Test Code 실행

`tcs/<app>/tc_<app>.sh`가 스펙의 절차를 그대로 셸 스크립트로 수행한다. 이때 판정에 쓴
명령은 실행만 하고 버리지 않고, `dump_cmd`로 결과를 그대로 남긴다.

```bash
# 나쁜 예 — 뭘 봤는지 아무도 검증할 수 없음
echo "파일이 생성되었음을 확인함"
assert "TC05-1: .xz 파일 존재" "PASS"

# 좋은 예 — 실제로 실행한 명령과 결과가 그대로 남음
dump_cmd ls -la "$LATEST_XZ"
assert "TC05-1: .xz 파일 존재" "PASS"
```

## ③ 근거 로그 3종 — SM, SL, 리눅스 명령어 결과

| 종류 | 무엇인가 |
|---|---|
| **리눅스 명령어 결과** | 스크립트가 판정을 위해 직접 실행한 `ls`/`du`/`xz --test` 등의 raw 출력 — "파일이 있다/없다" 같은 상태 확인 |
| **[SL] 로그** | `system_log` 애플리케이션 자신이 journald에 남기는 내부 동작 기록 |
| **[SM] 로그** | host 명령을 대신 실행해주는 `system_manager` 데몬이 journald에 남기는 실행 기록 |

리눅스 명령어 결과가 "결과 상태"를 보여준다면, `[SL]`/`[SM]`은 "그 결과가 나오기까지
애플리케이션 내부에서 무슨 일이 있었는지"를 보여준다. 셋이 서로 다른 각도에서 같은 사실을
증명해야 판정을 신뢰할 수 있다.

## 예시 — TC01 실제 결과 문서

`tcs/system_log/tc_system_log_result.md`에 세 근거가 이렇게 함께 남아 있다:

```markdown
| 기준 ID | 결과 |
|---------|------|
| TC01-1: 파일명 형식 `systemlog_{14자리}_{14자리}.log.xz` | **PASS** |

**근거 (tc_run.out):**
[PASS] TC01-1: 파일명 형식 (systemlog_시작_저장.log.xz)

**근거 (journald):**
[16:13:38][SL] [get_log_name] log_file_name: systemlog_20260709161334_20260709161338.log
[16:13:38][SM] Executing host command: xz -f .../systemlog_20260709161334_20260709161338.log
→ exit_code:0
```

`[SL]`은 "이 이름으로 파일을 만들기로 했다"는 내부 결정, `[SM]`은 "압축 명령이 실제로
실행돼 성공했다"는 실행 기록이다. 이 둘과 `ls -la`로 확인한 실제 파일 존재가 서로 맞아야
TC01이 진짜로 PASS라고 말할 수 있다.

## 원칙 요약

- "확인했다/되었다" 같은 말은 근거가 아니다 — 실행한 명령과 출력을 그대로 남긴다.
- 리눅스 명령어 결과(상태) + SL/SM 로그(내부 동작)를 함께 봐야 "우연히 맞은 결과"와
  "의도대로 동작해서 맞은 결과"를 구분할 수 있다.
- 스펙의 PASS/FAIL 기준, 스크립트의 판정, 결과 문서의 근거는 항상 서로 맞아떨어져야 한다.

브라우저 대시보드(`tools/tc_dashboard`)로 실행하면 이 대조(스펙 대비 결과, 명령 결과,
관련 SL/SM 로그)를 자동으로 묶어서 보여준다.
